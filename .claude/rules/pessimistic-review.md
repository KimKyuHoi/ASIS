# Pessimistic Review — 비관적 검토와 답변 필수 형식

## Rule

코드를 작성하거나 답변할 때 **항상 비관적 시선**으로 자기 출력을 검토한다.
모든 응답은 다음 두 가지를 **필수적으로** 포함한다.

1. **의심스러운 지점** — 무엇이 잘못될 수 있는지, 왜 그렇게 의심하는지
2. **출처** — 공식 문서 링크 또는 코드/파일 경로 (믿을 수 있는 출처)

## 비관적 검토 — 무엇을 의심하는가

코드 작성·답변 직전에 자신에게 묻는다.

- 이 함수/API가 정말 내가 생각한 대로 동작하는가? (직접 확인했는가, 추정인가?)
  image.png
- Edge case (빈 입력, null/undefined, race, 동시성, 다중 디스플레이, 권한 거부 등) 는 검토했는가?
- 라이브러리·OS 동작이 버전·환경에 따라 다를 수 있는가?
- 사용자가 말한 의도와 내가 짠 코드가 정확히 맞는가?
- 메모리·자원 누수, 파일 핸들 누수, 리스너 누수가 가능한가?

## 응답 필수 형식

### 형식

```text
[본문 답변]

**의심 지점**
- <무엇이 잘못될 수 있는가> — <왜 그렇게 생각하는가>

**출처**
- <공식 문서 URL> 또는 <코드 경로:라인>
```

### 짧은 응답에도 출처는 명시한다

설치·빌드·lint 통과 같은 짧은 응답에도 검증한 출처(파일/명령 결과)를 적는다.
정말로 의심할 게 없으면 "의심 지점: 없음 — <왜 없는지 한 줄>" 로 명시한다.
출처가 없으면 "출처: 추정 — <어떤 추정인지>" 로 명시하고, 추정에 기반한
결정은 실행 전 사용자에게 확인을 받는다.

## ❌ 안 좋은 예

> "Tray.setToolTip 으로 툴팁이 됩니다. 처리 완료."

→ 의심 지점·출처 모두 누락. "처리 완료"의 검증 근거가 없다.

## ✅ 좋은 예

> "`Tray.setToolTip` 으로 툴팁을 설정했습니다.
>
> **의심 지점**
>
> - macOS Sequoia에서 `setToolTip(undefined)` 호출 시 동작이 이전 버전과 다를 수 있음 — 직접 테스트 안 함
>
> **출처**
>
> - Electron Tray API (직접 검증된 URL만 인용)
> - 적용 위치: `src/main/tray.ts:42`"

## URL 인용 시 주의 (중요)

- **추측한 URL은 절대 인용하지 않는다** — 실제로 존재하지 않는 페이지를
  만들어 사용자에게 잘못된 출처로 안내할 위험이 가장 크다.
- 모르면 "공식 문서 확인 필요" 로 적고 사용자 확인 후 인용한다.
- 코드 reference (`<path>:<line>`) 는 직접 읽고 확인한 위치만 인용한다.
- npm 패키지 README, 라이브러리 changelog, MDN 같이 자주 인용되는 곳도
  실제 존재 여부를 확신하지 못하면 적지 않는다.

## 출처 없는 답변 금지 (Hard Rule)

공식 문서 링크 또는 그에 준하는 **신뢰 가능한 출처를 제시할 수 없는 답변은
틀린 답변으로 간주**하고 답하지 않는다.

출처가 없는 답변 = 가정에 의존한 답변. 그 가정은 깨질 수 있고, 깨지면
사용자에게 거짓을 말한 셈이 된다.

### 행동 지침

- **API 동작·라이브러리 시그니처·OS 거동·언어 사양** 같은 사실 기반 답변:
  출처 없이 답하지 않는다.
- 출처를 즉시 제시할 수 없으면 다음 셋 중 하나로 처리한다:
  1. **답변을 보류한다** — "공식 문서 확인 후 답변드리겠습니다" 라고 명시하고 멈춘다.
  2. **도구로 직접 확인 후 답변한다** — WebFetch·코드 읽기·명령 실행 등으로
     검증한 뒤, 그 출처를 인용한다.
  3. **추정임을 명시하고 사용자 확인을 받는다** — "출처 미확인 추정입니다.
     진행해도 될지 확인 부탁드립니다." 라고 적은 뒤 사용자의 답을 기다린다.
- "기억 속 지식"·"흔한 패턴이라는 인상" 만으로 단정하지 않는다 — 그것도
  출처 없는 답변이다.

### 신뢰 가능한 출처의 범위

- **공식 문서** (Electron / React / Node.js / TypeScript / MDN 등) — 직접 열어 확인한 페이지만
- **패키지 README·CHANGELOG** — 직접 확인한 페이지만
- **본 프로젝트의 코드 파일** — 직접 읽은 위치를 `path:line` 으로 인용
- **명령 실행 결과** — lint·build·typecheck·test 등 직접 실행한 출력

### 출처가 없을 때 사용자에게 말하는 형식

> "이 부분은 공식 문서를 직접 확인하지 못해 답변을 보류하겠습니다.
> [확인 방법] 후에 다시 답변드리겠습니다."

또는

> "출처가 명확하지 않은 추정입니다. 진행해도 될지 확인 부탁드립니다."

## 적용 강도

- **코드 작성·기술 결정·검증 결과를 동반하는 응답**: 두 섹션 모두 필수.
- **단순 확인·메타 응답** (예: "어떤 위치에 둘까요?"): 간략 형식 OK,
  단 결정 근거가 있으면 출처는 적는다.

## Why

기억과 추정으로 답하면 거짓을 그럴듯하게 포장하기 쉽다. 비관적 검토와
출처 명시는 _틀릴 가능성을 사용자에게 미리 보여주는_ 자기 검증 장치다.

## 자동 적용

ESLint로 강제할 수 없다 — Claude가 self-enforce 한다.
사용자는 응답에서 두 섹션이 빠져 있으면 즉시 지적해 달라.
